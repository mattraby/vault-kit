# vault-kit Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Package the Promethean OKF/LLM-wiki vault convention and its ingest skill as a reusable Claude Code plugin plus a clone-able template, so a new project can stand up the same standard in seconds.

**Architecture:** One canonical home — the plugin directory `plugins/vault-kit/` — contains the `new-vault` scaffolder (with the vault skeleton bundled inside it) and the generalized `ingest-source` skill. A build script regenerates a committed, clone-able `template/` at the repo root from those canonical sources, for users who clone without Claude Code. A repo-root marketplace manifest makes the whole repo installable.

**Tech Stack:** Markdown + YAML frontmatter (OKF v0.1), Claude Code plugin/marketplace JSON manifests, Bash scripts (verification + template build), Python 3 stdlib (`extract_pptx.py`, copied unchanged), Obsidian config JSON.

## Refinements discovered during planning (vs. the spec)

These supersede the spec where they differ; the spec's intent is preserved.

1. **Skeleton lives inside the plugin, not only at repo root.** Installed plugins are copied to an isolated cache (`~/.claude/plugins/cache`); a skill cannot reach files outside its plugin. So the canonical vault skeleton lives at `plugins/vault-kit/skills/new-vault/skeleton/`, referenced from the skill via `${CLAUDE_SKILL_DIR}/skeleton/`. The repo-root `template/` is a *generated, committed export* for the clone path.
2. **One build script, not a one-way sync.** `scripts/build-template.sh` regenerates the entire repo-root `template/` (vault skeleton + a copy of `ingest-source`) from the canonical plugin sources. This replaces the spec's `sync-ingest.sh` and also keeps the skeleton in sync.
3. **Plugin skills are namespaced.** Skills are invoked as `/vault-kit:new-vault` and `/vault-kit:ingest-source` (confirmed against current Claude Code docs), not bare `/new-vault`. `ingest-source` also auto-invokes from its description.
4. **`.obsidian/` is minimized to `app.json`.** Only the link-format settings are load-bearing for OKF portability; Obsidian regenerates the rest on first open. This avoids shipping dangling references to a theme/plugin we don't bundle.

## Global Constraints

Every task's requirements implicitly include these:

- **No engagement specifics.** `grep -ri promethean template/ plugins/` MUST return nothing. (The `docs/` spec/plan legitimately discuss Promethean origins and are excluded from this check.)
- **OKF conformance.** Every non-reserved `.md` in a vault has YAML frontmatter with a non-empty `type:`. Reserved files `index.md` and `log.md` have NO frontmatter. `CLAUDE.md` is schema/config and is exempt.
- **Links are relative markdown links between `.md` files.** No `[[wikilinks]]` anywhere in a vault (except `CLAUDE.md`, which documents the rule by quoting it).
- **Bundled scripts are referenced via `${CLAUDE_SKILL_DIR}/...`** so they resolve at plugin, project, and personal install levels.
- **Plugin manifest path is `plugins/vault-kit/.claude-plugin/plugin.json`; marketplace manifest is `.claude-plugin/marketplace.json` at repo root.**
- **`extract_pptx.py` stays Python 3 stdlib-only** (no pip installs).
- **Donor for verbatim copies:** the existing Promethean repo at `/Users/matt/Development/bolo/promethean` (`vault/` and `.claude/skills/ingest-source/`). It exists locally on this machine.

### Task 1: OKF conformance checker (`scripts/check-okf.sh`)

Build the verification harness first — later tasks use it as their test. It checks a vault directory for OKF conformance.

**Files:**
- Create: `scripts/check-okf.sh`

**Interfaces:**
- Produces: a CLI `scripts/check-okf.sh <vault-dir>` → exit 0 + `OK: ...` when conformant; exit 1 + `FAIL: ...` lines otherwise. Consumed by Tasks 2, 4, 5, 7.

- [ ] **Step 1: Write the checker**

Create `scripts/check-okf.sh`:

```bash
#!/usr/bin/env bash
# check-okf.sh — verify a directory is a conformant OKF / LLM-wiki vault bundle.
# Usage: scripts/check-okf.sh <vault-dir>
set -uo pipefail

VAULT="${1:?usage: check-okf.sh <vault-dir>}"
fail=0

# 1. Every non-reserved .md must have YAML frontmatter with a non-empty type:
while IFS= read -r f; do
  case "$(basename "$f")" in
    index.md|log.md|CLAUDE.md) continue ;;
  esac
  if [ "$(head -n1 "$f")" != "---" ]; then
    echo "FAIL: $f missing YAML frontmatter (no leading ---)"; fail=1; continue
  fi
  if ! awk 'NR==1{next} /^---/{exit} /^type:[[:space:]]*[^[:space:]]/{found=1} END{exit !found}' "$f"; then
    echo "FAIL: $f frontmatter has no non-empty 'type:'"; fail=1
  fi
done < <(find "$VAULT" -name '*.md' -type f)

# 2. Reserved index.md / log.md must NOT start with frontmatter
while IFS= read -r f; do
  if [ "$(head -n1 "$f")" = "---" ]; then
    echo "FAIL: reserved $f must not have YAML frontmatter"; fail=1
  fi
done < <(find "$VAULT" \( -name 'index.md' -o -name 'log.md' \) -type f)

# 3. No [[wikilinks]] anywhere (CLAUDE.md quotes the rule, so exempt it)
while IFS= read -r f; do
  if grep -Iq '\[\[' "$f"; then
    echo "FAIL: wikilinks ([[...]]) found in $f"; fail=1
  fi
done < <(find "$VAULT" -name '*.md' -type f ! -name 'CLAUDE.md')

if [ "$fail" -eq 0 ]; then echo "OK: $VAULT is OKF-conformant"; fi
exit "$fail"
```

Run: `chmod +x scripts/check-okf.sh`

- [ ] **Step 1: Write the build script**

Create `scripts/build-template.sh`:

```bash
#!/usr/bin/env bash
# build-template.sh — regenerate the clone-able template/ from canonical plugin sources.
# template/ is a committed, generated artifact: never edit it by hand; edit the plugin and re-run.
set -euo pipefail
ROOT="$(cd "$(dirname "$0")/.." && pwd)"
SKILLS="$ROOT/plugins/vault-kit/skills"

rm -rf "$ROOT/template"
mkdir -p "$ROOT/template/.claude/skills"

cp -R "$SKILLS/new-vault/skeleton" "$ROOT/template/vault"
cp -R "$SKILLS/ingest-source"      "$ROOT/template/.claude/skills/ingest-source"

cat > "$ROOT/template/README.md" <<'EOF'
# Knowledge Vault (OKF / LLM-wiki starter)

A starter knowledge vault: an Obsidian-compatible, portable **Open Knowledge Format (OKF) v0.1**
bundle maintained in the **LLM-wiki** style. This directory is a generated export of
[vault-kit](https://github.com/mattraby/vault-kit) for people who want the template without
installing the Claude Code plugin.

## Use it

1. Copy this directory to your project (e.g. `degit mattraby/vault-kit/template my-project`).
2. Open `vault/` in [Obsidian](https://obsidian.md).
3. Read `vault/CLAUDE.md` — the schema and workflows.
4. Rename the placeholder wiki categories; delete the two worked examples once the pattern is clear.
5. To ingest sources, the `ingest-source` skill is bundled in `.claude/skills/` for Claude Code.

> Generated by `scripts/build-template.sh` in vault-kit — do not edit here; edit the plugin source.
EOF

echo "Built template/ from canonical plugin sources."
```

````markdown
# vault-kit

Start a new **OKF / LLM-wiki knowledge vault** in seconds. vault-kit packages a proven
convention — an Obsidian-compatible, portable [Open Knowledge Format (OKF) v0.1](https://cloud.google.com/blog/products/data-analytics/how-the-open-knowledge-format-can-improve-data-sharing)
bundle maintained in the [LLM-wiki](https://gist.github.com/karpathy/442a6bf555914893e9891c11519de94f)
style — as a Claude Code plugin and a clone-able template.

## What you get

- **`/vault-kit:new-vault`** — scaffolds a conformant `vault/` (schema, `sources/` + `wiki/` with
  placeholder categories, worked examples, Obsidian config) into the current repo.
- **`/vault-kit:ingest-source`** — turns an external document (PPTX, PDF, Markdown, URL) into
  cross-linked vault knowledge, with a human review checkpoint. Also auto-invokes on requests
  like "ingest this deck."

## Install (Claude Code)

```
/plugin marketplace add mattraby/vault-kit
/plugin install vault-kit@mattraby
```

Then, in any project: `/vault-kit:new-vault` and start ingesting.

## Use without Claude Code

```
npx degit mattraby/vault-kit/template my-project
```

`template/` is a ready-to-use vault with the `ingest-source` skill bundled under `.claude/`.

## Repo layout

- `plugins/vault-kit/` — the plugin (canonical source).
  - `skills/new-vault/` — scaffolder; bundles the vault skeleton in `skeleton/`.
  - `skills/ingest-source/` — the ingest skill + `extract_pptx.py`.
- `template/` — generated, committed export of the skeleton + ingest skill (for the clone path).
- `scripts/check-okf.sh` — verifies a vault is OKF-conformant.
- `scripts/build-template.sh` — regenerates `template/` from the plugin sources.

## Maintaining

The plugin under `plugins/vault-kit/` is the single source of truth. `template/` is generated —
never edit it by hand. After changing the skeleton or the ingest skill, run:

```
scripts/build-template.sh
scripts/check-okf.sh template/vault
```

and commit the regenerated `template/`.
````
